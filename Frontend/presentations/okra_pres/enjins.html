Creating a claims engine for Regtech company

By James Twose

Change to business process

Assumptions

  • The company has access to API's of different airlines
  • The API's of the different airlines output data not in a standard format
  • The company has access to a database of all flights of the different airlines

Approach

  • Data engineering pipeline (e.g databricks, airflow)
  • Standardized data storage (e.g. postgresql, mongodb)
  • Train, test, deploy monitor model (e.g. AWS sagemaker, clarify)
  • User interface for users to interact with the output (e.g. PowerBI)
  • Overall Idea - semi automated ML process

Added Value

How does this help you?

  • reduced workload for the legal team (time, money)
  • increased likelihood of success of claims that are pursued (money)
  • increased customer satisfaction due to speeding up the process (money, time)
  • increased customer satisfaction due to increased likelihood of success based on previous claims (money)

Challenges

  • missingness in the data (e.g. not all airlines have the same data)
  • data quality (e.g. reliability of the data may vary per airline)
  • data security (e.g. how is the data stored, is it encrypted, who has access to it)
  • data privacy (e.g. GDPR compliance)
  • data storage (e.g. how much data is there, how is it stored, how is it accessed, can it be scaled)

ML Architecture

The exact architecture of the ML pipeline will depend on the infrastructure currently used by the company and can be ammended accordingly.

  • Data Engineering Pipeline - DataBricks
  • Data Storage - Postgresql
  • Model Training - AWS Sagemaker
  • Model Deployment/ Monitoring - AWS Sagemaker/ Clarify
  • User Interface - PowerBI
